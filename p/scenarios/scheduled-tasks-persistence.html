Scheduled Task Persistence Case Study

End-to-end purple-team lifecycle for Windows persistence via Scheduled Tasks: baseline detection and IR triage, EchoSentinel detection engineering (high-confidence signals + pivots), remediation/cleanup, and a validation retest to confirm the control outcome.

Purple Team Persistence Scheduled Tasks Detection Engineering Incident Response Remediation Control Validation EchoSentinel Windows Event Logs

Key Findings

  • Persistence value: scheduled tasks blend with legitimate admin automation; detection must focus on creation context and execution behavior.
  • Detection reality: task creation alone can be noisy; confidence comes from chaining logon + privilege + file write + task creation + forced execution.
  • Remediation: remove the malicious task, remove associated scripts/binaries, and harden who can create tasks + where they can write.
  • Outcome: EchoSentinel provides an explainable timeline and clean pivots from alert → raw evidence.

Detection Design

Signal Chain (High Confidence)

Scheduled tasks are common. Detection should focus on “who created it, from where, what it launches, and whether it runs immediately.”

  • Logon context — remote/admin logon signals anchoring the timeline
  • Privilege context — special privileges assigned to logon
  • Artifact creation — script/binary written prior to task creation
  • Task created — new scheduled task persistence entry
  • Forced run — task executed on demand (high intent signal)

Why This Works

  • Builds confidence through correlation instead of single noisy events.
  • Creates an analyst-friendly flow: alert → pivot to creation → pivot to execution → pivot to file artifacts.
  • Remediation is measurable: task and artifacts removed, retest confirms change in outcome.
  • Evidence-first structure makes it portfolio-grade: each claim has a screenshot anchor.
